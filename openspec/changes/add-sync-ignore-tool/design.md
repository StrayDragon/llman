## Context

**当前状态**
- Cursor 使用 `.cursorignore` 文件，采用 gitignore 风格的模式语法
- Claude Code 使用 `.claude/settings.json` 文件，在 `permissions.deny` 数组中使用 `Read()` 规则
- 两种格式无法互操作，开发者需要手动维护两份配置
- 两个工具都会自动遵守 `.gitignore`，因此转换范围仅限于工具特定规则

**约束条件**
- 不引入新的外部依赖（使用现有的 `inquire`、`serde_json`、`anyhow`）
- 必须支持试运行模式以预览更改
- 必须在修改用户文件前进行安全检查
- 遵循现有的工具命令模式和 x 子命令转发模式

**利益相关者**
- 同时使用 Cursor 和 Claude Code 的开发者
- 需要在项目间同步忽略规则的团队

## Goals / Non-Goals

**Goals:**
- 提供命令行工具在 Cursor 和 Claude Code 忽略格式之间进行双向转换
- 支持自动检测源格式和智能默认输出路径
- 提供交互式模式以简化用户体验
- 支持试运行和详细输出以确保安全
- 通过 x 子命令提供便捷的快捷方式

**Non-Goals:**
- 不支持 `.gitignore` 转换（工具已自动遵守）
- 不支持 Codex 或其他 AI 工具
- 不支持复杂的否定模式转换（Cursor 的 `!pattern` 将跳过并警告）
- 不修改现有的工具命令行为

## Decisions

### 1. 枚举类型用于格式表示

**决策:** 使用 `enum IgnoreFormat { Cursor, ClaudeCode }` 而不是字符串

**理由:**
- 编译时类型安全
- 更好的错误处理
- 更清晰的代码意图
- 易于扩展（未来可能添加其他格式）

**替代方案:** 使用字符串常量
- 拒绝原因: 运行时错误风险更高，IDE 支持较差

### 2. 模式转换函数分离

**决策:** 为每个方向创建独立的转换函数

```rust
fn cursor_to_claude_code(patterns: &[String]) -> Vec<String>
fn claude_code_to_cursor(patterns: &[String]) -> Vec<String>
```

**理由:**
- 单向转换逻辑更清晰
- 易于测试
- 便于未来添加其他格式转换

**替代方案:** 单个通用转换函数
- 拒绝原因: 逻辑复杂，难以维护

### 3. JSON 解析使用 serde_json

**决策:** 使用 `serde_json` 解析和修改 Claude Code settings.json

**理由:**
- 项目已有依赖
- 保留 JSON 格式（注释、缩进）虽然重要但不关键（settings.json 通常由程序生成）
- 类型安全的序列化/反序列化

**风险:** 可能丢失 JSON 文件中的注释
**缓解措施:** Claude Code settings.json 通常不包含手写注释

### 4. 文件自动检测顺序

**决策:** 按优先级顺序查找源文件

```rust
// 优先级顺序
.cursorignore          # Cursor
.claude/settings.json  # Claude Code
```

**理由:**
- `.cursorignore` 通常是用户首先创建的
- 避免意外读取错误的文件

### 5. 否定模式处理

**决策:** 跳过 Cursor 的否定模式 (`!pattern`) 并发出警告

**理由:**
- Claude Code 的 `permissions.deny` 不支持否定模式
- 静默跳过可能导致意外的安全风险
- 警告让用户知道发生了什么

## Risks / Trade-offs

### 风险 1: JSON 文件损坏

**风险:** 修改 settings.json 可能导致格式错误或数据丢失

**缓解措施:**
- 使用 `serde_json` 进行解析和序列化
- 写入前验证 JSON 结构
- 考虑使用原子写入（`atomic_write_with_mode`）
- 在试运行模式下显示预览

### 风险 2: 模式语义差异

**风险:** 两种格式的通配符行为可能不完全相同

**缓解措施:**
- 文档中明确说明转换限制
- 对于复杂模式发出警告
- 提供详细输出模式显示实际转换结果

### 风险 3: 用户体验不一致

**风险:** 用户可能期望完全自动化的一键同步

**缓解措施:**
- 提供交互式模式引导用户
- 试运行模式默认启用或强烈建议
- 清晰的错误消息和警告

### 权衡: 简洁 vs 完整性

**权衡:** 否定模式和某些高级通配符将不支持

**决策:** 优先覆盖常见用例（90% 的模式），而非边缘情况

**理由:** 简化实现，减少维护负担，警告用户不受支持的模式

## Migration Plan

**部署步骤:**
1. 添加新模块 `src/tool/sync_ignore.rs`
2. 更新 `src/tool/command.rs` 添加新命令
3. 添加 x 子命令转发
4. 更新 i18n 字符串
5. 添加集成测试
6. 更新文档

**回滚策略:**
- 新功能是增量的，不影响现有命令
- 如有问题，可通过移除新命令快速回滚
- 不涉及数据迁移或配置格式更改

## Open Questions

1. **问:** 是否需要支持 `.cursorindexingignore`（仅索引忽略）？
   **答:** 暂不支持，可在未来版本根据需求添加

2. **问:** 是否需要支持合并现有规则而非覆盖？
   **答:** 是，双向模式应智能合并，避免覆盖现有规则

3. **问:** JSON 文件写入时是否保留原有格式（缩进、尾随逗号）？
   **答:** 使用 `serde_json::to_string_pretty` 保持一致的格式，但可能丢失原始缩进风格

4. **问:** 是否需要验证输出路径是否在项目根目录？
   **答:** 是，应检查路径合理性，避免写入意外位置
